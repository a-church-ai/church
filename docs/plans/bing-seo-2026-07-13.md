# Bing SEO — resolve 37 flagged errors + prevent regression

**Date**: 2026-07-13
**Trigger**: Bing Webmaster Tools review of achurch.ai — 37 errors, all "Moderate" severity, across 3 rules.
**Prior context**: [`search-discoverability-2026-06-10.md`](search-discoverability-2026-06-10.md) shipped the per-page dynamic SSR that this plan verifies + hardens.
**Scope note**: **IndexNow is out of scope** — user decision. Sitemap-based re-crawl signals are used instead.

---

## Executive summary

Post-audit, the diagnosis flipped from the first draft of this plan. The 37 errors are ~35 stale-crawl symptoms of one root cause + 2 real content bugs:

**Root cause of the ~35 duplicate/identical errors**: prod's `/ask/[slug]` route returns **HTTP 200 for any URL**, including garbage. Verified live: `https://achurch.ai/ask/xyz-fake-slug-999` returns 200 with the SSR fallback title and description. That means every misspelled URL, discovered garbage URL, or historical stale crawl becomes another "duplicate" data point Bing counts — and the error class can never fully clear until the leak is closed.

**Root cause of the 2 "short desc" errors on `/privacy` and `/terms`**: descriptions are 70 and 61 chars — genuinely under Bing's 150-160 target.

**One structural fix eliminates the whole error class** (return 404 for missing conversations). One content fix resolves the remaining two. One inline assertion prevents regression. **No new scripts, no new tooling, no feature gates** — greenfield-appropriate.

Total active work: **~45 minutes**. Then a 3-5 day Bing re-crawl cycle to confirm.

## Fixes — reordered by leverage after audit

### 🔴 Fix #0 — Return 404 for missing `/ask/[slug]` (eliminates the whole error class)

**Scope**: one route handler, [`app/server/index.js` around line 267-343](../../app/server/index.js). Delete the SSR fallback branch entirely.

**Change**:

```js
app.get('/ask/:slug', async (req, res) => {
  const slug = req.params.slug.replace(/[^a-zA-Z0-9_-]/g, '');
  const messages = await loadConversation(slug);
  if (!messages || messages.length === 0) {
    return res.status(404).type('text/plain').send('Not found');
  }

  const meta = buildConversationMeta(messages);
  const html = await fs.readFile(
    path.join(__dirname, '../client/public/conversation.html'),
    'utf8'
  );
  // ... existing substitution chain, unchanged
  res.set('Content-Type', 'text/html').send(rendered);
});
```

**What this deletes**:

- The `if (meta) { ... }` guard (unnecessary — meta always exists now because we 404'd already)
- The silent `res.send(html)` fallback that ships the template raw
- Both hardcoded fallback strings' meaningful use (they only survive as template defaults, which now can only appear if someone reads the raw template file directly)

**What this changes for Bing**:

- Garbage URLs like `/ask/xyz` return 404 → Bing drops them from index over the next crawl cycle
- Every URL that DOES return 200 went through `buildConversationMeta()` → unique title + unique description guaranteed
- The "identical descriptions" and "identical titles" error classes become structurally impossible to recur

**What this changes for humans**:

- Someone hitting a broken/deleted conversation URL sees a 404 instead of an empty broken-looking page. Cleaner failure mode.
- If UX benefits from a friendlier 404 page later, that's a separate ~5-line change (create `404.html`, use `sendFile`); doesn't affect the SEO fix.

**Verification**: `curl -sI https://achurch.ai/ask/xyz-fake` → expect `HTTP/2 404` post-deploy.

**Target length**: 150-160 chars. Substantive content, not fluff.

**Proposed rewrites** (both verified in-length):

### 🟡 Fix #2 — Force sitemap re-fetch in Bing Webmaster Tools (leverage existing dynamic sitemap)

**Scope**: no code changes. One click in Bing WMT.

**Mechanism**: Bing WMT → Sitemaps → resubmit `https://achurch.ai/sitemap.xml`. Bing will re-crawl the entire sitemap on its next cycle.

**Why this replaces the "submit 12 URLs manually" approach from the plan's first draft**: the dynamic sitemap at [`app/server/index.js:440-533`](../../app/server/index.js) already emits `<url>` entries for all 482 `/ask/[slug]` pages with `<lastmod>` derived from file mtime. Bing has this list. A single sitemap re-fetch signals Bing to re-scan every URL in it — no need to submit 12 URLs one at a time.

**Additional lever available if needed**: touch the mtime on the flagged conversation JSONL files (or bump their content) so `<lastmod>` in the next sitemap generation is newer than Bing's last known value. Higher-priority re-crawl signal. Not needed for the initial submission; keep in reserve if the sitemap resubmit alone doesn't clear the flags.

**Verification**: Bing WMT → Sitemaps tab → look at "Last crawled" timestamp advance after resubmission. Then wait 3-5 days and re-check Recommendations.

### 🟢 Fix #3 — Inline SSR-meta assertion in `buildConversationMeta` (regression prevention, no new tooling)

**Scope**: ~5-line change in [`app/server/lib/utils/page-meta.js`](../../app/server/lib/utils/page-meta.js) around line 74.

**Change**:

```js
function buildConversationMeta(messages) {
  const firstQ = messages.find(m => m.role === 'user');
  if (!firstQ) return null;
  // ... existing logic ...
  const description = answer
    ? truncateAtWord(answer, 158)
    : truncateAtWord(question, 158);

  // Fail-loud on descriptions that would trigger Bing's short-desc rule
  // or look empty in a SERP snippet. Non-throwing so the page still ships,
  // but the warning surfaces in prod logs for grep-based auditing.
  if (description.length < 50) {
    logger.warn('SSR meta description below 50 chars', {
      slug: messages[0]?._meta?.slug,
      length: description.length,
      preview: description
    });
  }

  return { title, ogTitle, description };
}
```

**Why inline instead of a script**:

- No test infra exists; adding one for one assertion is disproportionate tech debt
- The failure mode we're guarding against (empty description) only manifests on real traffic; a scheduled script would miss most causes
- Structured `logger.warn()` is grep-able and shows up in the same log stream as everything else
- If we ever add a proper test runner, this assertion moves into a test cleanly — it's just a function-level invariant

**What this replaces**: the original plan's Fix #4 "add `app/scripts/verify-ssr-meta.sh`". The script approach was borrowed from a pattern that doesn't quite fit here — the existing `app/scripts/*.js` files are one-off content generators, not runtime verifiers.

**Verification**: grep prod logs for the warning message after a week. Zero warnings = healthy. Any warnings = specific slugs to investigate.

### ~~🟢 Fix #4 — DRY up the fallback strings~~ (retracted after implementation review)

**Retracted.** The first-draft rationale ("the fallback strings become dead code after Fix #0") was wrong. Those strings in `app/server/index.js:295-322` are the **first arguments to `.replace()` calls** — they need to exactly match the template defaults in `conversation.html` so the SSR substitution works. Deleting them would break every substitution in the SSR chain.

A cleaner refactor exists: change the template to use unique placeholder markers (e.g. `<!--SSR_TITLE-->`, `<!--SSR_DESC-->`) instead of repeating full HTML tag strings on both sides. That's a legitimate greenfield improvement but out of scope for this SEO fix — it touches the `/reflections/[slug]` route with the same pattern, and the shape of the change is larger than "delete the strings."

Ship Fix #0 without touching the fallback strings. Revisit as its own refactor if the duplication becomes a real maintenance issue.
